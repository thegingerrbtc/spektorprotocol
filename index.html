Spektor Protocol White Paper

Abstract

The Spektor Protocol introduces a novel approach to decentralized, self-custody wallet management by enabling secure and consistent access to digital assets without requiring users to store sensitive seed phrases or private keys. Instead of relying on stored recovery phrases, Spektor uses deterministic key derivation by combining static blockchain anchor points, user-defined personal data, and a unique user-designated PIN to generate high-entropy keys in-memory during each session. Importantly, keys are never stored on the user's device; they exist only temporarily during active sessions and disappear upon logout.

Spektor offers three deployment models—Lite, Self-Hosted, and Community Node—to cater to different user preferences regarding security, decentralization, and control. The Community Node model allows users to host their own nodes, creating a distributed network that enhances scalability and user autonomy without depending on centralized infrastructure.

1. Introduction

Problem Statement

Traditional wallet security models require users to store private keys or seed phrases to access their assets, posing significant risks. Many users store these sensitive phrases insecurely—using digital notepads, screenshots, emails, or text messages—making them vulnerable to loss or theft. Alternatively, relying on custodial services like exchanges introduces trust dependencies and potential privacy compromises, as users depend on third parties to safeguard their assets.

Spektor’s Deterministic Approach to Reliable Key Derivation

Spektor addresses these challenges by offering a self-custody model that eliminates the need for users to manage or store seed phrases. Spektor's logic leverages reproducible entropy sources—including static blockchain data, user-defined personal answers, and a secure PIN—to securely reconstruct the user's key in each session. This ensures consistent and reliable access without storage or retrieval concerns.

Benefits of Reliable, Deterministic Derivation

By combining stable blockchain data, user-specific inputs, and a secure PIN, Spektor generates high-entropy keys with precision. This deterministic approach allows users to access their wallets safely without needing to store sensitive recovery phrases, enhancing both security and user experience.

2. Deterministic Key Derivation Logic

At the core of Spektor's security model is deterministic entropy generation, which consistently and securely produces a unique key for each user session without persistent storage.

Entropy Generation

Spektor combines multiple secure sources:

  1. Blockchain Anchor Points: Incorporates immutable, verifiable blockchain data, such as the Ethereum genesis block hash, providing a trusted anchor without requiring network calls.
  2. User-Defined Personal Anchors: Users answer five personal questions (e.g., "What was your favorite childhood toy?"). Answers are standardized to ensure consistency, adding unique, user-specific entropy.
  3. PIN-Selected Hashes: A six-digit PIN further personalizes entropy by selecting specific blockchain anchors, enhancing randomness and security.

Consistent Key Derivation Logic

Spektor deterministically combines these entropy sources to produce the same secure key in every session. This reliable reproducibility allows users to confidently access their assets without needing stored data.

3. Session-Based Key Handling

Session-Based Key Generation

Spektor derives keys directly in server RAM at the start of each session. Keys exist only temporarily—"like a ghost"—during active use and are cleared upon logout. This minimizes exposure to potential security risks associated with persistent storage or the user's own imperfect storage solutions.

Avoidance of Persistent Storage

With on-the-fly derivation, there is no need to store private keys or recovery phrases, eliminating risks of data being compromised over time. Each session remains independent and secure.

Security Benefits of In-Memory Handling

By handling keys only in memory, Spektor reduces the attack surface. Sensitive key material exists solely in RAM during the session, and attackers would need real-time access to device memory to compromise it. This design isolates each session, preventing cumulative risks.

4. Deployment Models

Lite

A straightforward, Spektor-hosted instance ideal for users seeking simplicity. Lite clients connect to a Spektor-hosted server that securely handles key derivation and memory-only access.

Self-Hosted

Users install and run Spektor locally on their own hardware, giving advanced users maximum control over key derivation and session handling while benefiting from Spektor's consistent, deterministic key reproduction.

Community Node

Allows users to operate their own nodes, decentralizing the Spektor infrastructure. In this model, each node:

  • Verifies Code Integrity: Nodes pass hashing checks to ensure they are running genuine, unaltered Spektor code.
  • Processes API Requests: Nodes handle API requests for Lite clients, distributing network load and enhancing scalability.
  • Earns Transaction Fees: Each node earns 20% of the transaction fees for processed requests, incentivizing reliable uptime and participation.

5. Security Model

Deterministic Key Derivation

Spektor ensures reliable access by combining blockchain anchors, user-provided answers, and a secure PIN, enabling consistent key regeneration without storing sensitive information. This removes the risks associated with key storage while ensuring secure access.

Code Hashing for Integrity Verification

The protocol enforces code hashing to verify each node's integrity, preventing unauthorized or modified versions from participating in the network. Only verified instances of Spektor can generate the correct key, securing the network against tampering.

Memory-Only Key Handling

By keeping the user's private key only in RAM during the session and never transferring it to the client device, Spektor minimizes the risk of exposure, even if the client device is insecure. All functions requiring key access occur in-memory on the server.

6. Economic and Incentive Model

Transaction Fee Allocation

In the Community Node model, nodes that process transaction requests receive 20% of the associated transaction fees. This incentivizes users to operate reliable nodes and participate in the network.

Benefits of User Participation

This model promotes decentralization and network resilience by distributing both network load and incentives. It allows Spektor to scale through user-hosted infrastructure rather than centralized servers, enhancing both reliability and scalability.

7. Technical Implementation Details

Entropy Management and Key Generation

Spektor's deterministic derivation combines stable blockchain data, user inputs, and PIN-selected anchors to ensure consistent entropy across sessions. This reliable entropy allows users to access their wallets securely without local storage.

Code Hash Verification

Each node verifies its code against a reference hash to confirm integrity. This step prevents unauthorized modifications and ensures that only verified instances of Spektor participate in the network.

API Aggregator

An API aggregator manages Lite client requests, routing them to the most reliable nodes. This optimizes load distribution, ensuring efficient access without central bottlenecks.

8. Use Cases and Applications

  • Decentralized Wallet Security: Spektor offers a secure solution for decentralized wallet management by eliminating stored seed phrases and reproducing keys on-the-fly.
  • Community-Driven Node Network: Users can contribute to the ecosystem by hosting nodes, reinforcing Spektor's commitment to decentralization.
  • Institutional Use: Spektor's secure, tamper-proof key derivation model suits institutions needing private key management without centralized storage or custody.
